# Migrate saleh.soy from pandoc/Makefile to SvelteKit

## Context

This is **Phase 1B** of a three-phase project to unify saleh.sh and saleh.soy onto a single modern stack:

- **Phase 1A** (a parallel doc in `qirh/sala_v2`): rewrite `saleh.sh` from Vue 2 to SvelteKit. Same domain, same Netlify project. No blog work.
- **Phase 1B** (this doc, this repo): rewrite `saleh.soy` from pandoc/Makefile to SvelteKit. Same domain, same Netlify project. No SPA work.
- **Phase 2** (later, in `qirh/sala_v2`): merge this repo into `sala_v2`, route blog under `/blog/*`, retire this repo.
- **Phase 3** (later): point `saleh.soy` at the unified site via a 301 forwarder Netlify project.

This document is the blueprint for **Phase 1B only**. An executing agent should not touch `qirh/sala_v2` or domain config.

Phase 2 implies this repo will be archived shortly after Phase 1B lands. **That is by design — the Svelte rewrite makes Phase 2's merge much cheaper, even though both phases are throwaway-ish individually.**

## Goal
Ship a SvelteKit replacement of the existing pandoc/Makefile site at `https://saleh.soy`, with full feature parity. Visitors should not be able to tell the rewrite happened.

## Scope

**In:**
- Replace pandoc + Makefile with SvelteKit + `mdsvex`.
- Port the 8 published posts and 1 page (about) — markdown source stays as-is wherever possible.
- Reproduce: post list page, individual post pages, RSS feed, sitemap, theme toggle, OG/meta tags, RSS auto-discovery `<link>`, `Link:` header.
- Preserve every existing URL: `/`, `/{slug}`, `/about`, `/rss.xml`, `/sitemap.xml`, `/robots.txt`, `/style.css` references etc. **Do not change post slugs.**
- Preserve `static/*` asset paths (`/moi.jpg`, `/alien_blue.ico`, `/spiderman/*`, etc.) — the post markdown references them with absolute paths.

**Out:**
- New posts. (`in_progress/` stays untouched.)
- Visual redesign — should look the same.
- Anything saleh.sh-related.
- Anything domain-related.
- Schema for frontmatter beyond `title` + `date` (the only two fields used today).

## Target stack

- **SvelteKit** with the `@sveltejs/adapter-static` adapter — output is fully static.
- **[`mdsvex`](https://mdsvex.com/)** for markdown processing. Each `posts/*.md` becomes a Svelte page.
- **Svelte 5** with runes. JS (no TypeScript) to keep the stack lean — current site has no TS.
- **No CSS framework.** Keep the existing 123-line `static/style.css` as-is, served from `/style.css` like today.
- **Theme toggle:** keep the existing 22-line `static/theme.js` as-is, included via `<svelte:head>` or a layout `<script src="/theme.js">`. Do not rewrite it as a Svelte component — it's a pure DOM toggle that already works.

### Post routing

Two viable layouts. **Pick layout A unless something forces B.**

- **Layout A (recommended):** keep `posts/*.md` as the source of truth, copy each to `src/routes/<slug>/+page.svx` at build time via a one-line npm script (`scripts/sync-posts.sh`). Author edits `posts/funny-week.md`, runs `npm run build`, sync script copies + transforms it. **Pro:** authoring path doesn't change. **Pro:** `src/routes/` stays one-source-of-truth for routing. **Con:** small build step.
- **Layout B:** use a single `[slug]/+page.svx` with dynamic markdown loading. **Pro:** one route file. **Con:** mdsvex's static analysis works less well, prerender list has to be hand-fed.

Layout A is what the file tree above assumes.

### Generated files: lifecycle and gitignore

Generated by `sync-posts.sh`, **never committed**:
- `src/routes/<slug>/+page.svx` for every post (one dir per slug, marked with a `.generated` sentinel).
- `src/routes/about/+page.svx` (also `.generated`-marked).
- `src/lib/pages-manifest.json`.
- `src/lib/posts-manifest.json`.

**Single source of truth for content stays at `posts/*.md` and `pages/*.md`.** Committing the generated routes would create noisy diffs on every author edit and make slug renames a multi-file operation. Ignoring them means a fresh checkout needs `npm run sync` (or `npm run dev`/`npm run build`, both of which trigger sync via the `predev`/`prebuild` hooks in Step 2.1) before the routing tree is materialized — standard for any build-step project.

`.gitignore` additions for these (in addition to `node_modules/`, `build/`, `.svelte-kit/` from Step 1.3):
```
src/lib/pages-manifest.json
src/lib/posts-manifest.json
src/routes/about/+page.svx
src/routes/*/+page.svx
src/routes/*/.generated
!src/routes/+page.svelte
```

The `*/+page.svx` glob covers all post slugs without enumerating them. The `*/.generated` line is required: without it, every generated route dir shows up in `git status` because the sentinel file inside is un-ignored, which marks the whole dir untracked. The `!` line is just for clarity — `.svelte` and `.svx` are different extensions, but the negation makes intent explicit if someone changes the pattern later.

### Frontmatter contract

Posts preserve their existing frontmatter exactly:
```
---
layout: post              # ignored — present for legacy compatibility, do not break it
title: Funny Week
date: 2023-05-21
---
```

`mdsvex` exposes frontmatter as the page's `metadata` export — `title` and `date` consumed by `PostLayout.svelte`, RSS, and the index page.

**Pages may have no frontmatter.** `pages/about.md` currently has *zero* frontmatter — the existing Makefile derives the page title from the slug. Do not add frontmatter to `about.md` (out of scope per "Do not edit any `.md` file content"). Instead, `PageLayout.svelte` must default `metadata.title` from the route slug (titlecased): `about` → `About`. `sync-posts.sh` emits page slugs into `pages-manifest.json` so the sitemap covers every `pages/*.md` file.

## Migration mapping

| Old (pandoc/Makefile) | New (SvelteKit) | Notes |
|---|---|---|
| `Makefile` (`make`) | `npm run build` | One-line replacement once SvelteKit is configured. |
| `templates/post.html` (chrome) | `src/routes/+layout.svelte` | Site chrome only: `<header>`, nav, theme bootstrap script, `<script src="/theme.js">`, `<link rel="stylesheet">`, favicon, RSS auto-discovery. Shared by every page. |
| `templates/post.html` (article) | `src/lib/PostLayout.svelte` (mdsvex layout) | Per-post body only: `<svelte:head>` with `<title>` + per-post OG tags from frontmatter, `<article>` wrapper, `<h1>`, date, back-link top + bottom. **Does not** re-emit the site header. |
| `templates/page.html` (article) | `src/lib/PageLayout.svelte` (mdsvex layout) | Per-page body only: `<svelte:head>` with `<title>`, no date, no back link. **Does not** re-emit the site header. |
| `templates/index.html` | `src/routes/+page.svelte` | The post list. Loops `posts.js` and renders `<li><time>{date}</time> <a href="/{slug}">{title}</a></li>`. |
| `scripts/build_index.sh` | `src/lib/posts.js` (sort logic) + `src/routes/+page.svelte` | Sort by `date` desc. |
| `scripts/build_rss.sh` | `src/routes/rss.xml/+server.js` | Same XML shape, including `guid isPermaLink="true"`. RFC822 dates: emit literal `+0000` suffix (not `GMT`) to match current output — JS's `toUTCString()` returns `GMT`, so format manually. Excerpts come from `posts-manifest.json` (emitted by `sync-posts.sh` at prebuild time) — not from compiled `.svx` imports, which expose only `metadata` + the rendered component, not raw markdown. |
| `scripts/build_sitemap.sh` | `src/routes/sitemap.xml/+server.js` | Same XML shape. Consumes `pages.js` + `posts.js` + homepage; no hard-coded page URLs. |
| `scripts/smoke_test.sh` | `scripts/smoke_test.sh` (rewritten) | Adapt to assert on the new `build/` output. |
| `posts/*.md` | `posts/*.md` + sync to `src/routes/{slug}/+page.svx` | Preserve markdown verbatim. The sync script can be `for f in posts/*.md; do slug=$(...); cp "$f" "src/routes/$slug/+page.svx"; done`. |
| `pages/about.md` | `src/routes/about/+page.svx` (or sync similarly) | |
| `static/*` | `static/*` | Verbatim copy. SvelteKit serves `static/` at the URL root, same as today. |
| `netlify.toml` build line | New `[build] command = "npm run build"`, `publish = "build/"`. | The current pandoc-curl line goes away — pandoc is no longer needed in CI. |
| `[[redirects]] /*/ → /:splat 200` | **Keep verbatim.** | Trailing-slash handling for old Hugo-era inbound links. |
| `[[headers]] Link: </sitemap.xml>; rel="sitemap"` | **Keep verbatim.** | |

## Execution order

Each step ends with a verification an agent can run. If verification fails, stop and surface the error.

### Step 1: scaffold SvelteKit alongside the existing build
1. Run `npx sv create svelte` (Skeleton, JS, no TS, no extras). The legacy `npm create svelte@latest` is deprecated. Install `@sveltejs/adapter-static`, `mdsvex`.
2. Configure `svelte/svelte.config.js`:
   - `extensions: ['.svelte', '.svx', '.md']`
   - Preprocess with `mdsvex({ extensions: ['.svx', '.md'], layout: { post: '...PostLayout.svelte', _: '...PageLayout.svelte' } })`.
   - `kit: { adapter: adapter({ pages: 'build', assets: 'build', precompress: false, strict: true }), prerender: { entries: ['*'] } }`.
3. Add `node_modules/`, `build/`, `.svelte-kit/` to `.gitignore` (currently only ignores pandoc artifacts). The generated-route entries from "Generated files: lifecycle and gitignore" are added in Step 8 since they only become relevant at cutover when paths are at repo root.
4. **Verify:** `cd svelte && npm run build` produces a `build/` with an `index.html`.

### Step 2: wire up posts and pages
1. Write `svelte/scripts/sync-posts.sh`. The script must work both before and after cutover, so resolve paths from anchors that are stable across phases:
   - **Source root** (where `posts/` and `pages/` always live): `ROOT=$(git rev-parse --show-toplevel)`.
   - **App root** (where `src/` lives): `APP=$(cd "$(dirname "$0")/.." && pwd)` — script is always under `<app>/scripts/sync-posts.sh`, so `$(dirname "$0")/..` is the app root in both phases.
   - For each `"$ROOT"/posts/*.md`: `mkdir -p "$APP/src/routes/<slug>"` and copy to `"$APP/src/routes/<slug>/+page.svx"`.
   - Same for `"$ROOT"/pages/about.md` → `"$APP/src/routes/about/+page.svx"`.
   - Pre-cutover, `ROOT` is the repo root and `APP` is `<repo>/svelte/`. Post-cutover, `ROOT == APP == <repo>`. Both paths work without edits at cutover.
   - **Also emit manifests**:
     - `"$APP/src/lib/posts-manifest.json"` containing `[{slug, title, date, excerpt}, ...]` for every post. Excerpt: first non-empty paragraph after the frontmatter terminator `---`, with markdown collapsed to plain text via shell (`sed`-strip `[link](url)` → `link`, drop `**`/`*`/`#` markers), joined with single spaces, cut to 280 chars. This is intentionally simpler than `pandoc -t plain` — modest excerpt drift is acceptable per Step 3.
     - `"$APP/src/lib/pages-manifest.json"` containing `[{slug, title}, ...]` for every page, where title comes from frontmatter if present or the titlecased slug otherwise.
   - **Wipe stale generated routes before copying.** Sync writes a sentinel `.generated` file into every directory it creates (`src/routes/<slug>/.generated`, `src/routes/about/.generated`). At the start of each run, `find src/routes -name .generated -execdir sh -c 'rm -rf "$PWD"' \;` clears any previously-generated dir. Routes without a sentinel (`+page.svelte`, `+layout.svelte`, `rss.xml/`, `sitemap.xml/`) are untouched. This handles slug renames and deletions without leaving zombie routes.
   - Wire to **both** `predev` and `prebuild` in `package.json`. Without `predev`, a fresh `npm run dev` produces no routes and no manifest. Define a single `"sync": "sh scripts/sync-posts.sh"` script and reference it from both: `"predev": "npm run sync"`, `"prebuild": "npm run sync"`. **Invoke via `sh`, not the bare path** — that way the script doesn't need an executable bit, which fresh checkouts on some platforms or extracted-from-tarball setups may not preserve. (Alternative: `chmod +x scripts/sync-posts.sh` and commit, but `sh scripts/...` is one less thing to forget.)
2. Write `svelte/src/lib/posts.js` and `svelte/src/lib/pages.js`: import the generated manifests and re-export sorted arrays. **Do not** use `import.meta.glob` to derive metadata — `glob('+page.svx', { eager: true })` gives compiled Svelte modules (`{ metadata, default }`), not raw markdown, so excerpts can't be recovered there. The manifests emitted by sync-posts.sh are the source of truth for RSS, sitemap, and the index page.
3. Write `svelte/src/routes/+layout.svelte` — owns *all* shared site chrome: `<header>` (`<a class="site" href="/">saleh.soy</a>`, nav link to `/about`, `<button id="theme-toggle">`), `<svelte:head>` with the inline theme-bootstrap snippet, `<script src="/theme.js" defer>`, `<link rel="stylesheet" href="/style.css">`, `<link rel="icon" href="/alien_blue.ico">`, `<link rel="alternate" type="application/rss+xml" ...>`. Renders `<slot />` inside `<main>`.
4. Write `svelte/src/lib/PostLayout.svelte` (mdsvex layout, **not** a Svelte page). Use Svelte 5 runes: `$props()` for mdsvex props, `$derived()` for computed title/date/url, and `{@render children?.()}` for page body. Owns *only* per-post body: `<svelte:head>` with `<title>{title} — saleh.soy</title>` and per-post OG tags from frontmatter (`og:title`, `og:url`, `og:image`, `og:type=article`, `og:site_name`, `twitter:card`); markup is `<article><h1>{title}</h1><p class="date"><time>{date}</time></p><p class="back"><a href="/">← all posts</a></p>{@render children?.()}</article><p class="back"><a href="/">← all posts</a></p>`. **Does not** emit `<header>` or theme scripts — the root layout handles those.
5. Write `svelte/src/lib/PageLayout.svelte` (mdsvex layout). Same split as PostLayout and same runes style: `<svelte:head>` with `<title>` (defaulting to titlecased slug if `metadata.title` is missing — see Frontmatter contract), and `{@render children?.()}` for the body. No date, no back link, no header.
6. Write `svelte/src/routes/+page.svelte` (homepage post list): `<svelte:head><title>blog</title><meta name="description" content="Saleh's blog"><meta property="og:type" content="website">...</svelte:head>` matching `templates/index.html` literally (current title is `blog`, not `saleh.soy`). Body: `<ul class="posts">{#each posts as p}<li><time datetime={p.date}>{p.date}</time> <a href="/{p.slug}">{p.title}</a></li>{/each}</ul>`.
7. **Verify:** `npm run dev`, visit `/` (post list shows 8 entries sorted by date desc, `<title>` is `blog`), `/funny-week` (post renders with title, date, body, single `<header>` from the root layout — not duplicated), `/about` (page renders, `<title>` is `About — saleh.soy`).

### Step 3: build the RSS endpoint
1. Write `svelte/src/routes/rss.xml/+server.js`. Reuse the exact XML shape from `scripts/build_rss.sh` (`<rss>`, `<channel>`, `<item>` blocks, and `<guid isPermaLink="true">`). Pull `{slug, title, date, excerpt}` from `src/lib/posts.js` (which re-exports `posts-manifest.json` from Step 2.1) — excerpts are precomputed by `sync-posts.sh`, not derived at request time. RFC822 dates: format manually as `Sun, 21 May 2023 00:00:00 +0000` (literal `+0000`, not JS's default `GMT`).
2. Set `Content-Type: application/xml`.
3. Mark the route prerendered: `export const prerender = true;`.
4. **Verify:** `npm run build && xmllint --noout build/rss.xml` parses without error; `<item>` count equals `posts/*.md` count; every item has non-empty `<title>`, `<link>`, `<guid>`, `<pubDate>`, `<description>`. **Do not** assert byte-for-byte parity with the current `public/rss.xml` — the existing excerpts come from `pandoc -f markdown -t plain` and replicating its quirks (link footnoting, smart quotes, ellipsis) without pandoc is out of scope. Modest excerpt drift is acceptable; RSS clients dedupe by `<guid>` (slug-based, stable).

### Step 4: build the sitemap endpoint
1. Write `svelte/src/routes/sitemap.xml/+server.js`. Loop `posts.js` + `pages.js` + the homepage. Same XML shape as `scripts/build_sitemap.sh`.
2. Mark the route prerendered: `export const prerender = true;`. (Without this it won't be in `build/`, since the sitemap isn't linked from any page — `prerender.entries: ['*']` only crawls reachable links.)
3. Set `Content-Type: application/xml`.
4. **Verify:** `xmllint --noout build/sitemap.xml` parses without error and `<loc>` count matches `1 + |posts| + |pages|`.

### Step 5: theme toggle
1. Confirm `static/theme.js` and `static/style.css` both exist (carry over from the current repo). The layout from Step 2 already includes them — no Svelte component needed.
2. **Verify:** in dev, click the toggle button (id `theme-toggle`), `data-theme` attribute on `<html>` flips, persists across reload.

### Step 6: OG, meta, favicon, RSS auto-discovery
1. Cross-check `<head>` against `templates/post.html` and `templates/index.html`. Items that must be present on every page:
   - `<link rel="stylesheet" href="/style.css">`
   - `<link rel="icon" href="/alien_blue.ico">`
   - `<link rel="alternate" type="application/rss+xml" title="saleh.soy RSS" href="/rss.xml">`
   - Theme bootstrap script (the `<script>(function(){try{var t=localStorage.getItem('theme');...})()</script>` snippet) inline in `<head>` so there's no flash.
   - `<script src="/theme.js" defer>`
   - Per-post: `og:title`, `og:url`, `og:image`, `og:type=article`, `og:site_name`, `twitter:card`.
2. **Verify:** view-source on `/funny-week`, all of the above present and equal to current production.

### Step 7: rewrite smoke tests
Assert against **served URLs**, not on-disk file paths. The current pandoc build emits `public/<slug>.html`; SvelteKit may emit `build/<slug>.html` or `build/<slug>/index.html` depending on trailing-slash config. Both serve `/<slug>` correctly through Netlify, but a smoke test that hard-codes a file shape would couple itself to adapter config that may change.

1. Rewrite `scripts/smoke_test.sh` to:
   - Run `npm run build` then `npm run preview -- --host 127.0.0.1 --port 4173 &` and wait for the port to be reachable.
   - `curl -fsS http://127.0.0.1:4173/<path>` for every URL invariant; trap-kill the preview server on exit.
2. Invariants asserted by curl:
   - `/` returns 200, `<title>` is `blog`, has 8 `<li>` entries each with `<time>` + `<a href="/<slug>">`.
   - `/<slug>` returns 200 for every post; body contains the post `<h1>`, `<time>`, and back-link `← all posts` at top and bottom.
   - `/about` returns 200; `<title>` is `About — saleh.soy`.
   - `/rss.xml` returns 200, an XML content type (`application/xml` from the endpoint or `text/xml` from the static preview server), parses with `xmllint --noout`, `<item>` count equals `posts/*.md` count.
   - `/sitemap.xml` returns 200, parses, `<loc>` count equals `1 + |posts| + |pages|`.
   - `/sitemap.xml` includes a `<loc>` for every `posts/*.md` and every `pages/*.md`, not just the correct count.
   - `/robots.txt` returns 200.
   - `/style.css`, `/theme.js`, `/moi.jpg`, `/alien_blue.ico` all return 200.
   - Per-post head: `og:title`, `og:url`, `og:image`, `og:type=article`, `og:site_name`, `twitter:card` all present.
   - `id="theme-toggle"` present on `/`, `/funny-week`, `/about`.
   - No regression: `<pre><code>&lt;img` does not appear in any rendered post (pandoc-era smoke check, kept).
3. **Verify:** `npm test` passes.

### Step 8: cutover
1. Move `svelte/*` to repo root, preserving the `scripts/` subdirectory, then wipe generated/installed dirs and remove `svelte/`:
   ```sh
   mv svelte/{package.json,package-lock.json,svelte.config.js,vite.config.js,src} .
   mv svelte/scripts/sync-posts.sh scripts/sync-posts.sh
   rm -rf svelte                # nukes node_modules/, build/, .svelte-kit/, empty scripts/
   ```
   Notes:
   - Brace expansion alone would put `sync-posts.sh` in the repo root — the explicit second `mv` keeps it under `scripts/` where `predev`/`prebuild` expect it.
   - `package-lock.json` exists from Step 1's `npm install` and must move with `package.json` to keep dependency resolution reproducible.
   - `rm -rf svelte` (not `rmdir`) handles `node_modules/`, `build/`, and `.svelte-kit/` left behind by Step 1's install + Step 2's dev runs. `rmdir` would fail on these.
2. Delete: `Makefile`, `templates/`, `scripts/build_index.sh`, `scripts/build_rss.sh`, `scripts/build_sitemap.sh`. Keep `scripts/smoke_test.sh` (rewritten in Step 7).
3. Update `netlify.toml`: replace the `command = "curl ... | tar ... && PATH=... make"` line with `command = "npm run build"`; change `publish = "public"` to `publish = "build"`. Keep the `[[redirects]]` for trailing-slash handling. Keep the `[[headers]]` Link header.
4. Update `package.json` scripts: `build`, `dev`, `preview`, `test` (calls smoke_test.sh), `sync` (calls sync-posts.sh), `predev` and `prebuild` (both call `npm run sync`).
5. Update `.gitignore` to include all entries from "Generated files: lifecycle and gitignore" plus `node_modules/`, `build/`, `.svelte-kit/`. Verify a sample run: `npm run sync && git status` — generated routes and manifests should be untracked.
6. **Verify:** `rm -rf node_modules build .svelte-kit && npm install && npm run build && npm test` all pass.
